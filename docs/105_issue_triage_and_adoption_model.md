# Issue Triage and Adoption Model

## Issue labels

| Label | Meaning |
|---|---|
| `good-first-issue` | Safe for new contributors |
| `docs` | Documentation improvement |
| `site` | Static site or GitHub Pages change |
| `schema` | JSON schema/sample data improvement |
| `backend` | FastAPI/service logic change |
| `security-boundary` | Requires careful review |
| `connector-governance` | Connector contract or governance work |
| `provider-governance` | Provider/model routing work |
| `accelerator` | Business accelerator demo work |
| `not-live-safe` | Proposed work touches live execution; blocked by default |

## Adoption funnel

1. Reader understands the narrative from README.
2. Reader opens GitHub Pages console.
3. Reader runs backend tests.
4. Reader runs the CLI demo.
5. Reader inspects one governance or policy flow.
6. Reader contributes a scenario, schema, doc, or deterministic service improvement.

## Public presence assets

- README narrative
- control-plane console
- procurement accelerator demo
- API catalog page
- contributor console
- release scorecards

## Monetization-adjacent but open-source-safe assets

- readiness assessment templates
- governance scorecards
- accelerator demos
- architecture review checklists
- workshop outlines

These support consulting and implementation work without turning the open-source project into a sales brochure.
