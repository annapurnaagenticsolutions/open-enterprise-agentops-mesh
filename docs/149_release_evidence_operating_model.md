# Release Evidence Operating Model

## Evidence categories

| Category | Evidence |
|---|---|
| Functional evidence | Backend tests, API smoke checks, benchmark harness |
| Public repo evidence | Required files, site pages, docs, OpenAPI-lite catalog |
| Demo evidence | Storyboard, recording checklist, public proof bundle |
| Safety evidence | Disabled live execution boundaries and no raw secret policy |
| Adoption evidence | Community intake, issue templates, contributor guidance |

## Evidence lifecycle

1. Generate or update assets.
2. Validate tests and smoke checks.
3. Run benchmark harness.
4. Run public repo validation.
5. Generate release evidence report.
6. Review demo recording checklist.
7. Publish or record only after all critical items are ready.

## Quality standard

A public release should not depend only on claims. It should have visible proof artifacts that a reviewer can inspect without running hidden systems.
